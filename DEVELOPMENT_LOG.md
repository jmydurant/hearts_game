# 开发日志

这个文档记录 `Terminal Hearts` 从原型到当前可玩版本的关键里程碑，以及开发过程中踩过的坑。

## 项目目标

- 做一个桌面端单机红心大战
- 风格是“窗口里的终端”，不是宿主 shell 里的 `ncurses` 程序
- 优先保证 `Windows / macOS / Linux` 的一致性
- UI 使用像素风和整数缩放，不接受模糊放大

## 开发阶段

### 阶段 1：确定技术路线

最初讨论过两条路线：

- 真实终端程序：`ncurses`
- 窗口程序里模拟终端外观：`Godot`

最后选 `Godot 4 + GDScript`，原因很直接：

- 跨平台一致性更强
- 不受不同终端、字体、颜色能力的影响
- 更容易控制像素风、全屏、导出和后续动画

这一步的结论基本决定了后面所有实现方式。

### 阶段 2：打通可玩的最小闭环

对应提交：`0e1d5d9`

这一阶段的目标不是美观，而是让一局牌能从头打到尾。

完成内容：

- 1 名玩家对 3 个 AI
- 标准红心大战基础规则
- 发牌、传牌、出牌、吃墩、结算
- `shoot the moon`
- 累计到 100 分结束
- 基础 headless 测试

这里的核心原则是：规则层和 UI 层分离。规则写在 `scripts/core/`，UI 只消费状态，不直接拥有规则。

### 阶段 3：把“能玩”提升到“能看”

对应提交：`7419938`

这一阶段主要在 `scripts/ui/game.gd` 上打磨：

- 调整布局，避免初版 UI 过于拥挤
- 引入像素字体
- 改善牌面花色辨识度
- 修方向键输入
- 修传牌阶段和出牌阶段的焦点/按钮行为
- 补 `README.md`

这一步的核心变化是把 UI 从“证明功能存在”推进到“可持续手玩验证”。

### 阶段 4：解决导出和资源加载问题

对应提交：`23d89a0`

Ubuntu 导出可执行程序后，运行时报错找不到外部字体文件：

`Can't open file from path 'assets/fonts/fusion/fusion-pixel-8px-monospaced-zh_hans.ttf'`

修复方式：

- 优先使用 `load("res://...")` 从 Godot 资源系统加载字体
- 仅在开发环境资源加载失败时，才回退到真实文件路径

这一步的教训很明确：Godot 导出后，资源路径语义和开发环境不是一回事，不能把本地文件路径方案直接搬到导出版。

### 阶段 5：修正全屏和分辨率行为

对应提交：`c21703e`

这一阶段主要处理两个问题：

- 初始化和切换全屏时，打印当前分辨率、viewport 和缩放倍率
- 修掉 `F11` 后只看到一个角的问题

这里最终确认的做法是：

- 日志里同时打印 `window / screen / viewport / canvas / scale`
- 布局计算基于 `viewport`，不是物理屏幕尺寸

这个调整之后，全屏显示链路才真正稳定下来。

### 阶段 6：从固定 `1920x1080` viewport 改成动态整数倍 viewport

这一阶段主要是为了解决 macOS 上的两个实际问题：

- 当前桌面可用区域明明足够大，但因为系统缩放和高分屏语义，启动时仍可能被判成“不支持”
- 切到最大化或全屏后，窗口尺寸变大了，内部逻辑画布却仍停留在 `1920x1080`，视觉上只是把旧画面继续放大

旧方案本质上是：

- 基础逻辑网格 `640x360`
- 根 viewport 固定 `1920x1080`
- 等价于把逻辑网格长期锁在 `3x`

这在 `1080p` 上没有问题，但在更大的窗口和全屏环境下，会暴露出两个缺点：

- 大屏额外空间没有真正转化成更多可用逻辑像素
- macOS 上的高分屏和缩放桌面，会让“必须先有 1920x1080 有效区域”这个门槛显得过于生硬

最终调整为：

- 把项目基础 viewport 改回 `640x360`
- 启动时根据当前可用区域决定初始整数倍率，优先 `3x`，必要时回退 `2x`
- 在窗口尺寸变化、最大化和全屏切换时，按当前窗口可容纳的最近整数倍动态更新 viewport
- 保留 `窗口化 / 最大化 / 全屏` 的显示模式切换

这一步的核心价值不是“让画面更大”，而是让逻辑网格和真实可用空间重新对齐。

结论：

- 像素风项目应该优先固定基础网格，而不是长期固定某一个放大后的 viewport 尺寸
- `1080p` 可以是一档推荐体验，但不应该再被写成所有桌面环境唯一合法的内部画布
- 对高分屏和全屏来说，正确做法是动态选择整数倍，而不是把旧画布继续强行拉伸

## 踩坑总结

### 1. 真实终端和“终端风格窗口”不是一回事

一开始如果继续走 `ncurses`，很多显示问题会直接变成“不同系统终端实现差异”。这类问题通常不值得为一个牌类游戏买单。

结论：

- 如果目标是风格像终端，优先自己画
- 如果目标是真在 shell 里运行，再选终端库

### 2. 逻辑分辨率过低，放大后一定会糊

早期逻辑画布过小，放大到 `1920x1080` 后颗粒太粗，中文尤其难读。

后来的调整是：

- 提高逻辑网格密度
- 正文字体和标题字体分级
- 拉开面板间距

结论：

- 像素风不等于分辨率越低越好
- 中文像素字体尤其需要给它留足空间

### 3. 中文像素字体不能照搬西文字号思路

早期系统字体虽然不糊，但气质不对；后来直接上像素字，又出现“字块太大、阅读累”的问题。

最后采取的是折中方案：

- 正文用 `Fusion Pixel 8px`
- 标题用 `Fusion Pixel 12px`

结论：

- 中文 UI 不适合全局强风格大字号像素字
- 标题和正文要分开处理

### 4. Godot 导出后不能依赖真实文件路径加载资源

这是最典型的导出坑之一。

开发环境里 `ProjectSettings.globalize_path()` 能工作，不代表导出包里也能工作。导出后资源通常已经进了 `PCK`。

结论：

- 游戏运行时资源优先走 `res://`
- 文件系统路径只做回退，不做主路径

### 5. 方向键被 GUI 焦点系统吞掉了

早期方向键失灵，不是逻辑没写，而是用了 `_unhandled_key_input()`，导致按键先被 Godot GUI 焦点导航吃掉。

修复方式：

- 改到 `_input()`
- 仅在确实处理了事件后再 `set_input_as_handled()`

结论：

- 键盘驱动型 UI 不要默认把输入交给 GUI 焦点系统

### 6. “合法牌”渲染状态不能直接复用当前状态机输出

传牌结束后但还没轮到玩家时，整手牌颜色会异常变灰。根因是把当前回合的 `legal_moves` 直接拿去染玩家手牌，但那时轮到的是 AI。

结论：

- UI 渲染条件必须和“当前是否轮到玩家”绑定
- 状态机输出是事实，不代表适合直接显示给用户

### 7. 传牌阶段的按钮可用性不能依赖 `current_player`

传牌阶段如果还拿 `current_player == 0` 作为按钮焦点前提，就会出现“按钮看起来不能选”的问题，因为该字段在传牌后续逻辑里服务的是首个出牌者，不一定是玩家。

结论：

- 不同 phase 的字段语义要分开看
- 同一个字段不要在多个 phase 里强行复用同一套 UI 判断

### 8. 花色颜色必须做到一眼区分

如果 `♠` 和 `♣` 共用类似灰白色，在快速扫牌时非常难受。

后来做法是：

- `♠` 改为偏深蓝
- `♣` 改为偏绿
- `♦` 和 `♥` 也拆成不同暖色
- 卡牌顶部增加细色带

结论：

- 牌类游戏的颜色设计，第一目标是辨识，不是克制

### 9. `viewport` 拉伸模式下，不能再按物理屏幕坐标摆 UI

这是全屏模式里最关键的坑。

项目使用的是 `stretch/mode="viewport"`。在这种模式下，Godot 会先把内部逻辑 viewport 处理好，再整体上屏。如果这时还拿 `DisplayServer.screen_get_size()` 去算根节点位置，就会把 UI 偏移到屏幕外，最后只剩一个角。

修复方式：

- 用 `get_viewport_rect().size` 做布局基准
- 把物理屏幕尺寸只用于日志观察，不再直接参与 UI 坐标计算

结论：

- 在有 stretch 的项目里，布局坐标和显示器物理尺寸必须严格分离

### 10. 高分屏“更大”和“更合理”不是一回事

4K 屏全屏时，把同一套 `640x360` 逻辑画布继续往上整数放大，确实会更大，但并不一定更好看。

曾经采用过的保守方案：

- `1080p` 保持当前体验
- 全屏限制最大舒适倍率

现在已经先往前走了一步：

- 基础网格仍是 `640x360`
- 运行时 viewport 会按当前窗口选择最近整数倍

更完整但还没做的方案是：

- 为 `1080p / 1440p / 4K` 建独立布局档位

结论：

- 高分屏不能只理解成“把现有像素继续放大”
- 真正的解法是多档布局 profile

## 当前结论

当前版本已经完成了这些关键能力：

- 规则可玩
- 导出可运行
- 中文像素 UI 基本可读
- 全屏切换可用
- 关键输入链路稳定

但它依然是一个原型，不是最终版本。后续最值得投入的方向仍然是：

- 多档高分屏布局
- 更强 AI
- 更细的动效和反馈
- 更完整的导出与发布流程
